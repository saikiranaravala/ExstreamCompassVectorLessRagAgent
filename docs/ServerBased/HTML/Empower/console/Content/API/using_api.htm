Using the Empower API

The Empower API is a REST interface that lets integrators use API calls to control functions such as importing documents to the Empower Editor interface, opening documents in the editor interface, and exporting documents for fulfillment. The documentation provided here assumes that you are familiar with the principles of software development, APIs, and HTTP protocols.

You can use the following topics to learn more about the Empower API:

API authentication

Access to the Empower API is defined in the customerSecurityContxt.xml file during installation, and is controlled by the authorization provider that you configure when you set up the Spring Security framework. By default, Empower supports the administrator, editor, and integrator authorities.

The API format

Every string that is passed in an Empower API request must be UTF-8 encoded. For best results, normalize all strings to the Unicode Normalization Form C (NFC) format.

Dates logged by Empower operations, such as the document import time and date, are set to Coordinated Universal Time (UTC/GMT).

The API request structure

The Empower API supports the HTTP DELETE, GET, and POST methods. All Empower API requests consist of a URL and a header. The body of certain POST requests also include a payload.

The Empower API request structure consists of the following elements:

  • Request URL
  • Request HTTP header
  • Request HTTP body

Request URL

Use the following format for request URLs:

METHOD https://<server>:<port>/<context>/resource/{resource-name}/{parameters}/{method-name}

For more information about the variables that are contained in the request URL for each Empower API call, see the Empower API documentation.

Request HTTP header

Security token header

Empower requires session-based security tokens for all POST and DELETE API calls. These session-based tokens help to prevent malicious applications or websites from performing unwanted actions through the web browser of a user who has been authenticated to the Empower server (a type of attack known as a Cross-Site Request Forgery—or CSRF—attack).

Before your integration application can make DELETE and POST API calls, it must use the Get security token API call to capture the token response parameters for the user's current session, and then use the header token name and value in the HTML header for the API calls.

To include the token in a request header, use the following format:

{csrfHeader}: {csrfToken}

For example:

X-CSRF-TOKEN: 95cf86a3-079f-4860-aa62-dd64f8a8c321

You can also view the security token by retrieving the XSRF-TOKEN cookie after making an API call from a session that is already in progress.

Note: Keep in mind that if a session expires or changes, you must acquire a new security token.

XML header

By default, the Empower API response body contains a JSON payload. To receive an XML response instead of a JSON response, use the following HTTP header in your request:

Accept: application/xml

Request HTTP body

Empower does not support including a body in GET or DELETE requests.

Some POST requests include the application/x-www-form-urlencoded payload type in the body of the request. The multipart/form-data format is used for POST requests that include file uploads.

The API response structure

The Empower API response consists of the following elements:

  • Response HTTP header
  • HTTP response status codes
  • Response HTTP body
  • Response examples
  • Empower API response messages

Response HTTP header

Only header objects that contain a value are returned in the response.

This header item Describes
Accept-Language

The language requested in the header (if available)

If the language that was requested in the header is not available, Empower returns the default language.
Cache-Control Whether or not the object is publicly accessible
content-disposition How the content is delivered
Content-Length The size of the response payload
Content-Type

The media type of the response payload

The Empower API supports JSON and XML payloads.
Date The date and time of the response
HTTP code The standard HTTP response code
Server The type and version of the Empower Server
Transfer-Encoding Whether or not transfer encoding has been applied to the message body

 

Example POST response HTTP header:

HTTP/1.1 200 OK
Cache-Control=private
Content-Type=application/json
Date=Mon, 02 Feb 2015 21:06:42 GMT
Server=Apache-Coyote/1.1
Transfer-Encoding=chunked

Example GET response HTTP header:

HTTP/1.1 200 OK
Cache-Control=no-store,no-cache,must-revalidate,max-age=0,s-maxage=0,post-check=0,pre-check=0
Content-Type=application/octet-stream

HTTP response status codes

Status code Description
200: OK The call succeeded and the requested content is being returned.
400: BAD REQUEST

The server was unable to understand the request. This error usually occurs for one of the following reasons:

  • There are invalid parameters
  • Mandatory parameters are missing
  • There is an invalid Accept HTTP request header
401: UNAUTHORIZED

The server was unable to authorize the request. This error usually occurs for one of the following reasons:

  • An authentication token is missing
  • An authentication token is invalid
  • An authentication token signature is invalid
  • An authentication token signature has expired
403: FORBIDDEN The user is logged in, but is not authorized to perform this action.
404: NOT FOUND The requested record is not available. This error can indicate that the record never existed or that it has been deleted.
405: METHOD NOT ALLOWED The method type specified in the request is not valid. For example, the request specified the GET method for a call that requires the POST method.
406: NOT ACCEPTABLE The specified Accept header is not supported for the call.
415: UNSUPPORTED MEDIA TYPE The requested media type is not supported for the call. For example, the call requests the application/x-www-form-urlencoded media type, when only the multipart/form-data media type is supported for this call.
500: INTERNAL SERVICE ERROR The server failed to fulfill the request and has encountered an error.

Response HTTP body

Empower API calls can return HTML, an Empower document, or a standard HTTP body. The default Empower API response body contains a JSON payload. To generate an XML response instead of a JSON response, use the Accept: application/xml HTTP header in your request.

JSON container schema

The following JSON schema defines the structure of the Empower API response body:

{
  "id": "/",
  "type": "object",
  "properties": {
    "header": {
      "id": "header",
      "type": "object",
      "properties": {
        "status": {
          "id": "status",
          "type": "object",
          "properties": {
            "code": {
              "id": "code",
              "type": "string"
            },
            "msg": {
              "id": "msg",
              "type": "string"
            }
          },
          "required": [
            "code"
          ]
        }
      },
      "required": [
        "status"
      ]
    },
    "body": {
      "id": "body",
      "type": "object",
      "properties": {
            This depends on the service's method
      }
    }
  },
  "required": [
    "header",
    "body"
  ]
}

XML container schema

The following XML schema defines the structure of the Empower API response body:

<xs:schema attributeFormDefault="unqualified" elementFormDefault="qualified" xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="ResponseContainer">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="header">
                    <xs:complexType>
                        <xs:sequence>
                            <xs:element name="status">
                                <xs:complexType>
                                    <xs:sequence>
                                        <xs:element type="xs:string" name="msg" minOccurs="0"/>
                                    </xs:sequence>
                                    <xs:attribute type="xs:string" name="code"/>
                                </xs:complexType>
                            </xs:element>
                        </xs:sequence>
                    </xs:complexType>
                </xs:element>
                <xs:element name="body" minOccurs="0">
                    <xs:complexType>
                        <xs:sequence>
                            <!-- This depends on the service's method -->
                            <xs:any processContents="lax"/>
                        </xs:sequence>
                    </xs:complexType>
                </xs:element>
            </xs:sequence>
        </xs:complexType>
    </xs:element>
</xs:schema>

Response examples

To see examples of JSON and XML responses for the API methods that Empower supports, see the Empower API documentation.

Empower API response messages

Empower response messages include information that you can use for debugging, such as the details of notifications or errors, as well as corresponding message codes.

For information about the API call that returns the Empower API response codes, see View response codes.

Empower API message examples

When you attempt to access a document that does not exist, the Empower API returns the following payload header:

{
	"ResponseContainer": {
		...
		"header": {
			"status": {
				"code": "BUSI110E",
				"msg": The following document could not be found: 19b0eb38-0091-48c4-a1a5-417773d103b7."
			}
		}
	}
}

When there are no notifications or errors, Empower API returns the following payload header, which contains a status code of 0, and no message:

{
	"ResponseContainer": {
		...
		"header": {
			"status": {
				"code": "0"
			}
		}
	}
}
