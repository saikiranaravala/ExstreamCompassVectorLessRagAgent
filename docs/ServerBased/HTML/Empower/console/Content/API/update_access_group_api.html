Update an access group

The POST API call for access groups edits the attributes of an access group.

Who can access this call

Access to the Empower API is defined in the customSecurityContext.xml file during installation, and is controlled by your authorization provider. By default, Empower supports the administrator, editor, and integrator authorities.

Authenticated users with the following authority can use this API call:

  • administrator

Request

Request method Content type Syntax
POST application/x-www-form-urlencoded /<context>/resource/accessGroups

The <context> variable is configured during Empower installation.

The {docId} is the ID of the existing document.

Request header

Header Value Use Description
X-CSRF-TOKEN {csrfToken} Required

Provides a session token that authorizes you to access the DELETE and POST API calls

To view the token for your current session, do one of the following:

  • To capture the csrfToken response parameter, see Get security token.
  • Retrieve the XSRF-TOKEN cookie after making an API call from a session that is already in progress.

Keep in mind that if a session expires or changes, you must acquire a new security token.
Accept application/xml Optional Requests that Empower return an XML response instead of the default JSON response

Request parameters

Parameter Data type Use Description

accessGroupName

String

Optional

Specifies the name of the access group.

The maximum string length is 250 characters.

This parameter supports the ^[\p{L}\p{M}\p{N}_\-\.\s]*$ characters.

interactiveRole

List <String>

Optional

Specifies an interactive role that you want to associate with an access group.

To associate more than one interactive role with an access group, include multiple interactiveRole parameters in the request.

Example:

interactiveRole=CSR1&interactiveRole=CSR2

Note: Interactive roles are created and maintained in Design Manager. For information about how to set up interactive roles, see "Setting up Role-Based Editing in Empower Documents" in .

Example request

Element Example

Request header

 POST http://sample-server.example.com:8080/mpw/resource/accessGroups
X-CSRF-TOKEN: 95cf86a3-079f-4860-aa62-dd64f8a8c321
Request body Content-Type: application/x-www-form-urlencoded;

accessGroupName=CustomerService&interactiveRole=CSR1&interactiveRole=CSR2

Response

Empower supports two formats for API response payloads: JSON and XML.

Example JSON response

Media type: application/json

{
    "header": {
        "status": {
            "code": "SERV100I"
        }
    },
    "body": {
      "accessGroup": {
        "accessGroupId": "a89c20ae-f2b8-42ef-b098-e36834470221",
        "accessGroupName": "CustomerService",
        "interactiveRoles": [
            "CSR1",
            "CSR2"
        ]
     }
   }
 }

Example XML response

Media type: application/xml

<?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<ResponseContainer>
    <header>
        <status code="SERV100I"/>
    </header>
    <body>
        <accessGroup accessGroupId="a89c20ae-f2b8-42ef-b098-e36834470221 accessGroupName="CustomerService"/>
            <interactiveRole>CSR1</interactiveRole>
            <interactiveRole>CSR2</interactiveRole>
        </accessGroup>
    </body
</ResponseContainer>

Response parameters

If a parameter object does not contain a value, a JSON response returns a null value for the object, but an XML response does not include the object.

Parameter Data type Description
accessGroupId String

The ID of the access group

This value is generated automatically by Empower.
accessGroupName String The name of the access group
interactiveRoles String

The interactive roles that are associated with this access group
