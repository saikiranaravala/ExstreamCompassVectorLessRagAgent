Get applications

The GET API call for applications returns a list of the Empower applications that match your request parameters.

Who can access this call

Access to the Empower API is defined in the customSecurityContext.xml file during installation, and is controlled by your authorization provider. By default, Empower supports the administrator, editor, and integrator authorities.

Authenticated users with any of the following authorities can use this API call:

  • administrator
  • integrator

Request

Request method Syntax
GET /<context>/resource/applications?{parameters}

The <context> variable is configured during Empower installation.

The {parameters} search variable is optional. If you do not include any parameters, the search returns a list of all Empower applications.

Request header

Header Value Use Description
Accept application/xml Optional Requests that Empower return an XML response instead of the default JSON response

Request parameters

You can use the sorting, filtering, searching, and page size parameters to limit the number of Empower applications that are returned in a response.

Sorting

Parameter Data type

Use

 Description
sortField String Optional

Returns a list of Empower applications that is sorted by the value of the specified application response parameter

The maximum string length is 50 characters.

This parameter supports the ^[A-Za-z]* characters.
sortOrder String Optional

Sorts the list of Empower applications in ascending or descending order

This parameter can have a value of ASC or DESC.

Note: If you use the sortField parameter without the sortOrder parameter, the list of applications is sorted in ascending order.

Filtering

To filter the results when you search Empower applications, specify values for one or more of the parameters in the following table.

You can use these tips to help you refine your search:

  • To exclude a string from a filter request, precede the string with a dash (-). When you precede a string with a dash, Empower excludes applications that match the string that follows the dash.

  • To look for a string that begins with a dash, add the backslash (\) escape character just before the dash in your search string. For example, docTag=\-documentname.

  • To narrow the filtering results, you can string multiple parameters together with an ampersand (&). For example, appID=task&accessgroupName=Name.
Parameter Data type Use Description

accessgroupName

String

 Optional

Returns a list of the Empower applications that have been assigned the specified access group

The maximum string length is 250 characters.

This parameter supports the ^[\\p{L}\\p{M}\\p{N}_\\-\\.\\*\\s"\\\\]*$ characters.

Note: For more information about how to search for document owners, see Using search operators.
appId String Optional

The application ID that was generated by Empower

The maximum string length is 43 characters.

This parameter supports the \p{XDigit}{1,10}-\p{XDigit}{1,10}-\p{XDigit}{1,10}-\p{XDigit}{1,10} characters.
applicationName String Optional

The name of the Exstream application that was used to design the document

The maximum string length is 250 characters.

This parameter supports the ^[\p{L}\p{M}\p{N}_\-\.\s]*$ characters.
applicationOI Number Optional

The Object Identifier (OI) of the Exstream application
applicationPath String Optional

The folder path of the Exstream application in your installation of Design Manager

The maximum string length is 250 characters.

This parameter supports the ^[\p{L}\p{M}\p{N}_\-\.\/\\\ ]*$ characters. 

Note: If the Exstream application is located in the root folder of your installation of Design Manager, the applicationPath field will be empty.
databaseId UUID Optional

The database ID that is stored in the Empower application

The maximum string length is 36 characters.

This parameter supports the \p{XDigit}{8}-\p{XDigit}{4}-\p{XDigit}{4}-\p{XDigit}{4}-\p{XDigit}{12} characters.
editorVersion String Optional

The version of the editor that is stored in an Empower application and that is assigned to all new documents

The maximum string length is 250 characters.

This parameter supports the ^[a-zA-Z0-9\.]*$ characters.
engineVersion String Optional

The version of the engine that was used to package the Exstream application

The maximum string length is 250 characters.

This parameter supports the ^[a-zA-Z0-9\.]*$characters.

packageCreationDate

Date Optional

The date that the package file was created

The date format must be YYYY-MM-ddTHH:mm:ss.
packageFile String Optional

The name of the Exstream application package file, including the folder path

The maximum string length is 500 characters.

This parameter supports the ^[\p{L}\p{M}\p{N}_\-\.\:\/\\\ ]*$ characters.

packageVersion

String

 Optional

The version of Exstream that was used to package the application

The maximum string length is 250 characters.

This parameter supports the ^[a-zA-Z0-9\.]*$ characters.
previewPubFile String Optional

The name of the Exstream package file that all new documents will use to generate PDF previews (when Engine as a Web Service is configured as the PDF preview provider)

The maximum string length is 250 characters.

This parameter supports the ^[\p{L}\p{M}\p{N}_\-\.\s]*$ characters.

Searching

You can search across a set of supported string fields.

Note: Empower does not support using the dash (-) operator to exclude a search term on the q parameter.

Parameter Data type

Use

 Description
q String Optional

Returns a list of the Empower applications that match the search value that is specified for any of the following indexed string parameter fields:

  • accessgroupName
  • appId
  • applicationName
  • applicationOI
  • applicationPath
  • databaseId
  • editorVersion
  • engineVersion
  • packageFile
  • packageVersion
  • previewPubFile

The maximum string length is 500 characters.

This parameter supports the ^[\\p{L}\\p{M}\\p{N}_\\-\\.\\*\\s"\\\\@]*$ characters. 

You can narrow your search results by structuring your search request and using operators. For more information about using search in the Empower API, see Using search operators.

Controlling how search results are displayed

When your repository contains a large number of Empower applications, it is important to consider how your integration application interface will display the results when users search for Empower applications. This API call supports several parameters that you can use to control how your integration application displays pages of results, and to help reduce the amount of time that it takes to return search results.

Tip: For best results, design your integration application to use searching, filtering, and pagination to return a single page of results.

Parameter Data type

Use

 Description
limit Long Optional

Sets a limit on the number of applications that Empower can return in the response (the page size)

Note: You must use the limit and offset parameters together.
offset Long Optional

Tells Empower how many results to skip before returning the first search result

Note: You must use the limit and offset parameters together.
maxTotal Long Optional

Tells Empower to count to the last application that is returned by the search, or to the maxTotal number of applications (whichever is larger)

You can use this parameter to decrease the response time for this API call, by limiting the number of applications that are counted during the repository search.

Note: If you use the maxTotal parameter without specifying the limit and offset parameters, Empower sets the default pagination values to limit=maxTotal&offset=0.

Example requests

The following examples show how to create various types of requests using some of the parameters that are supported by this API call. Keep in mind that this API call does not support using a request body.

Example 1:

Request a list of all Empower applications with no pagination.

Element Example

Request header

 GET http://sample-server.example.com:8080/mpw/resource/applications

Example 2:

Request a single Empower application with a specified application ID (if the application exists).

Element Example

Request header

 GET http://sample-server.example.com:8080/mpw/resource/applications?appId=51-1115057666-1531728594-0

Example 3:

Request a list of all Empower applications that have been assigned a specified version of the editor, sorted in ascending order by the date that the package was created, and displaying only the first two applications.

Element Example

Request header

GET http://sample-server.example.com:8080/mpw/resource/applications?editorVersion=1.1.0.7783&sortField=packageCreationDate&limit=2&offset=0

Example 4:

Request a list of all Empower applications, with pagination that returns applications 51-100 and that counts all of the applications in the repository.

Element Example

Request header

GET http://sample-server.example.com:8080/mpw/resource/applications?limit=50&offset=50

Notice that this example uses only the limit and offset parameters. Depending on the size of the applications repository, not including the maxTotal parameter with the limit and offset parameters can result in long response times, because Empower counts the total number of applications in the repository. However, using only the limit and offset parameter lets you provide your users with the flexibility to go to any page in the results, up to and including the last page.

When you use this request example, the integration application interface displays the results on page 2, and lets users move forwards or backwards in the search results, or go to a specific page in the results:

Example 5:

Request a list of all Empower applications, with pagination that returns documents 101-150 and that counts to the last application that is returned.

Element Example

Request header

GET http://sample-server.example.com:8080/mpw/resource/applications?limit=50&offset=100&maxTotal=0

In this example, specifying a maxTotal value of 0 lets you minimize the response time for large application repositories, because Empower counts only to the last application that is returned. When you view the response for this request, you can look at the exceedsTotal response parameter to determine whether additional matching applications exist.

When you use this request example, the integration application interface displays the results on page 3, and lets users move forward in the results one page at a time, or select any of the previous pages in the results:

Example 6:

Request a list of all Empower applications, with pagination that returns applications 101-150, and that counts only 450 applications.

Element Example

Request header

GET http://sample-server.example.com:8080/mpw/resource/applications?limit=50&offset=100&maxTotal=450

In this example, specifying a maxTotal value of 450 lets you decrease the search response time in large application repositories, because Empower counts only the first 450 applications that match the search parameters. When you view the response for this request, you can look at the exceedsTotal response parameter to determine whether additional matching applications exist.

When you use this request example, the integration application interface displays the results on page 3, and lets users move forwards or backwards in the search results, or go to a specific page in the results up to page 10:

Response parameters

If a parameter object does not contain a value, a JSON response returns a null value for the object, but an XML response does not include the object.

Parameter Data type Description
accessgroupNames String The names of the access groups that control access to documents that are associated with the Empower application
appId String The application ID that was generated by Empower
applicationName String The name of the Exstream application that was used to design the document
applicationOI Number The Object Identifier (OI) of the Exstream application
applicationPath String

The folder path of the Exstream application in your installation of Design Manager

Note: If the application is located in the root folder of your installation of Design Manager, the applicationPath field will be empty.
databaseId UUID The database ID stored in the Empower application
editorVersion String The version of the editor that is assigned to all new documents
engineVersion String The version of the engine that was used to package the Exstream application
exceedsTotal Boolean The response parameter that indicates whether the total number of Empower applications that match your request exceeds the value that is returned for the total response parameter
limit Long

The maximum number of Empower applications that the request could have returned (the page size)

If the value of the limit response parameter is 0, then no limit was used, and the response included all of the applications that matched your request parameters.
offset Long The number of results that were skipped before the first result was returned

packageCreationDate

Date

The date that the package file was created

The date format is YYYY-MM-ddTHH:mm:ss.
packageFile String The name of the Exstream application package file, including the folder path

packageVersion

String

 The version of Exstream that was used to package the application
previewPubFile String The name of the Exstream package file that all new documents will use to generate PDF previews (when Engine as a Web Service is configured as the preview provider)
total Long

The total number of Empower applications that match your request parameters

If your request includes the maxTotal parameter, Empower counts to the last Empower application that is returned by the search, or to the maxTotal number of applications (whichever is larger).
