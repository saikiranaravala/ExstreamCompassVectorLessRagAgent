Known issues

Note: The issues that are reported here are not intended as statements of future support or limitations. Some issues might be related to the design of a particular application and therefore will not affect you. Some issues are defects that have already been corrected, and will be available in a future maintenance release of the software. Some issues are expected behavior, and therefore will not change in future maintenance releases. To find out whether these issues have been resolved, see the version-specific release notes or maintenance notes that accompany each release.

If you are using a container design that requires a frame that is not used in the standard design, create a separate document or page for your container design, and configure the page so that it is used only for container designs (In Design Manager, on the Interactive tab of the page properties, the Use for container designs only check box).

Empower Editor Editable areas If you apply text rules that partially cover text paragraphs in editable areas (paragraph objects, table cells, and text boxes), then end users might see unexpected behavior in Empower Editor.

You can apply text rules that fully cover text paragraphs in editable areas, or you can apply text paragraph rules to editable areas (in Design Manager, on the Targeting tab of the paragraph properties, the Rule section; or in Designer, from the right-click menu, Add text rule, or on the Rule tab of the paragraph properties).
Empower Editor Embedded interactive areas When you place embedded edit areas in an automated table row, end users cannot make changes to them.

Empower does not support embedded interactive areas in automated rows. If you need to include editable content in an automated table row, you must ensure that you use editable variables instead of embedded interactive areas.
Empower Editor  Flow frames In an Empower document, if there is too much content to fit in a content frame, and the next available content frame is not large enough to hold the overflow content, then the overflow content will flow to the next available content frame of sufficient size on a flow page. However, when end users preview the content or send the Empower document to the engine for fulfillment, the overflow content does not appear in the third frame as expected. This issue occurs because Empower allows content to skip frames where overflow will not fit, but the engine does not. When the engine encounters the first flow frame where the overflow content will not fit, it does not continue looking for additional frames where the content can fit. If you design an Empower document with flowing content, place the flow frame that will accept overflow content adjacent to the frame where the overflow comes from.
Empower Editor Flow frames End users cannot select editable content because an adjacent flow frame overlaps it.

Do not place editable content near a flow frame that can overlap it. For example, if you place an editable autosized text box adjacent to a flow frame, and an end user makes changes that cause the text box to expand and overlap the flow frame, Empower Editor places the flow frame on top of the editable content. In that scenario, end users cannot click to access the editable content that is beneath the flow frame.
Empower Editor Flow frames If your Empower document contains a page with relative flowing objects, and you place a flow frame on the same page, if you set the destination of overflow to Copy this page (in Design Manager, in the properties for the page, on the Flow tab, in the Destination of overflow from this page drop-down list), end users might see unexpected behavior in Empower Editor.

Design a separate flow page to capture overflow from the originating page. Then, in Design Manager, on the properties for the originating page, set the overflow destination to the separate flow page.
Empower Editor Flow frames If your Empower document contains relative objects in flow frames, the pagination in Empower Editor differs from that of engine output. When Empower Editor determines whether a flow frame is wide enough to hold an object, it evaluates only the current object being placed. The engine, on the other hand, evaluates the entire flow list and disables any flow frames that are not wide enough to hold objects in the flow list.

When designing a flow list that contains relative objects, make sure that flow frames are wide enough to hold all of the objects in the flow list.
Empower Editor Formatted text variables If your Empower document contains formatted text variables that are arrays, formatting changes that end users make to an array element are also applied to the array element that follows. If you need to show all elements of a formatted text variable that end users can format, use the All, paragraphs setting when you configure the variable in Designer (on the Variable Use tab of the variable properties, the Array element to use list).
Empower Editor Formatted text variables Formatted text variables that are hyperlinks do not work in Empower Editor.

You cannot apply a hyperlink to a formatted text variable in Designer; however, you can import hyperlinks in formatted text variables in Empower by using reference files for RTF content, and web services for HTML content.
Empower Editor Formatted text variables If your design uses rules to compare the values of formatted text variables to determine which content to include or exclude in your Empower document, the engine does not include the correct content in your Empower document.

No workaround. Do not use rules to evaluate formatted text variables in your Empower documents.
Empower Editor Formatted text variables If your Empower document contains multiple instances of a plain text-formatted text variable, any formatting changes that end users make to a single instance of the variable are propagated to every other instance of the variable.

No workaround is necessary—this is expected behavior. If you need to limit the ability of end users to make formatting changes to text in an Empower document, do not use editable formatted text variables.
Empower Editor Formatted text variables If your Empower document uses formatted text variables that are arrays, and you set the value of the array element to Automatic; All, print as list; or All, blanks (in Designer, in the variable properties, on the Variable Use tab, in the Array element to use drop-down list), then only the first element in the array appears in the Empower document.

In order to make all elements in a formatted text variable array appear in an Empower document, set the value of the array element to either All, paragraphs or All, line breaks.
Empower Editor Formatted text variables If your Empower document uses formatted text variables that are arrays to populate the values of a drop-down list, the values in the drop-down list do not appear as expected—specifically, Empower Editor replaces the values with a reference code.

No workaround. Do not use formatted text variables that are arrays to populate the values of drop-down lists in your Empower documents.
Empower Editor Formatted text variables If your Empower document uses more than one instance of a formatted text variable that contains multiple paragraphs, if end users delete a paragraph from one of the instances, they might see unexpected behavior in Empower Editor.

No workaround. If you need to let end users delete text from an Empower document, do not use editable formatted text variables.
Empower Editor Formatted text variables If a list uses a formatted text variable array to populate its options, end users receive an error when they try to select an option.

As a result, you should increase the size of flow frames to ensure that they are large enough to accommodate rounding differences on flowing tables.

Empower Editor Variables (custom formatting)

If your Empower document contains variables that use custom formatting, end users see unexpected results for the following combinations of variable types and custom format strings:

  • Float and currency variables—If you use the 0 and # characters in combination to specify digits and characters that should be dropped, Empower Editor populates variable values with digits in all cases. For example, if you specify a custom string of 0.00#, Empower Editor populates a variable value of 1.23 with 1.230.

  • String variables—If you use the & character (to specify a character or a space) and the @ character (to specify a character or nothing) in combination, Empower Editor does not populate variable values as expected.

No workaround. Avoid using the specified combinations of variable types and custom format strings (in Design Manager, on the variable properties, on the Output Format tab, the Custom format string box; and in Designer, on the variable properties, on the Variable Use tab, with Custom selected in the Special Formatting drop-down list, the box below the drop-down list).
Empower Editor Variables (output formatting)

When you apply output formatting to an editable variable in Designer or Design Manager (in Designer, the Special Formatting drop-down list on the Variable Use tab of the variable properties; in Design Manager, the Output format drop-down list on the Output Format tab of variable properties), while end users edit the variable in an Empower document, Empower Editor shows the actual variable value, without applying any formatting settings.

For example, suppose that you want the value for a variable to appear in all caps, so you apply the Upper, Keep Blanks setting in Designer or Design Manager. Even if the actual variable value is abcd, Empower Editor initially populates the variable with the correctly formatted value of ABCD. When an end user clicks the editable variable area, however, Empower Editor changes the variable to its actual value (abcd), and then allows the end user to enter any value (including lower-case). When the end user selects an area outside of the editable variable area, Empower Editor once again applies the correct (upper-case) formatting to the variable value.

This issue is cosmetic only and does not affect fulfillment or final output. The engine honors all output formatting for variables when creating output from edited Empower documents.

Important: This can be resolved by upgrading to version 16.4.6 or 16.4.10 of the Exstream engine.

Empower Services Layer Clustering (before version 16.4)

In versions of Empower earlier than 16.4, it is possible for the index on an Empower cluster node to become corrupted if, for example, a synchronization operation is interrupted or there is an unplanned power outage. If an index becomes corrupted, end users who attempt to access a resource in the affected repository will receive a 500-level error, and the application server log for the node will include a warning message describing the failed operation.

Important: This workaround should not be used with Empower 16.4 or later.

You must re-index the contents of the repository on each node in the cluster.

To recreate all of the indexes for a node:

  1. Stop the Empower application.
  2. In the empower.home directory for the node, delete the index folder.
  3. Restart the node.
  4. Repeat step 1 through step 3 for each node in the cluster.

If an index is missing an update, use one of the following the Empower API calls to update the indexes:

To re-index all of the repositories for a node:

http://<server>:<port>/
<EmpowerContext>/console
/tools/repository
/reindex.jsp?sync=true

To re-index only editors:

http://<server>:<port>/
<EmpowerContext>/console
/tools/repository
/reindex.jsp?
sync=true&path=/lib

Empower Services Layer Performance If Engine as a Web Service (EWS) and Empower are installed on the same server and EWS is configured as the preview provider for Empower, performance issues can occur during heavy, concurrent loads (for example, when an Empower Editor end user opens multiple previews in the same editing session).

As a best practice, do not run EWS and the Empower Services Layer on the same server.
Empower Services Layer Performance When Empower is installed on an IBM WebSphere 8.5.5.7 application server, a memory leak that occurs on some WebSphere 8.5.5.7 servers could affect the performance of the Empower application. If a WebSphere memory leak occurs, Empower administrators and Empower Editor end users might notice that Empower operations take longer than expected.

To reduce the amount of memory used by the IBM StAX parser, try setting the internal buffer size to 2048 bytes in the WebSphere server settings.

On a WebSphere application server, the internal buffer size is controlled by the following property:

com.ibm.xml.xlxp2.api.util.
encoding.DataSourceFactory.
bufferLength

For more information about configuring WebSphere properties, see the WebSphere documentation.
Empower Services Layer Performance

In versions of Empower earlier than 16.6, it is possible for the repository database to become corrupted if you use the default database connection pool settings.

If you are running Empower 16.6.0 or later, using the default database connection pool settings could result in degraded performance for Empower Editor users.

Set the maximum number of connections for your database connection pool to 100, and the minimum number of connections to 10.

For information about configuring connection pool parameters, see the documentation for your application server.
Empower Services Layer Spring Security permissions Permissions that are added to the customSecurityContext.xml file after the "denyAll" entry fail with a 403 error.

When you make changes to the customSecurityContext.xml file, do not add permissions after the "denyAll" string:

<!-- Error pages -->
<intercept-url pattern="/error/**" access="permitAll" />
<!-- Deny all other url's -->
<intercept-url pattern="/**" access="denyAll" />
