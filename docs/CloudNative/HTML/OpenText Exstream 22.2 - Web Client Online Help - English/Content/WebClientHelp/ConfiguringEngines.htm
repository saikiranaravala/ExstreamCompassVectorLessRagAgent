Configuring Exstream engines

On the Engines panel in System management view, you can manage the Exstream engines that are available for all tenants in your deployment. You can upload, configure, and delete engines to allow for easier control over orchestration, testing, and engine upgrades. Additionally, you can name the engines in your deployment so that users can more easily distinguish between the different engines available in your deployment.

Uploading engines

At least one engine must be available in the system tenant to generate previews, run simulations, and produce output in your Exstream environment. You can also upload additional engines as needed for specific applications. Keep in mind that Exstream engine runs are initiated from specific domains within each tenant, so the behavior of an engine run depends on the domain type and the product key used to license the tenant.

As you upload engines, keep the following considerations in mind:

  • Exstream engine ZIP files are available for download on My Support.

  • Exstream deployments support only 64-bit Linux engines, version 20.2 or later.

Note: To upload engines, you must be a member of the otadmins group in the OTDS system tenant.

To upload an engine:

  1. From My Support, download the ZIP file for the Exstream engine that you want to upload.

  2. Sign in to the Exstream web client as an OTDS administrator user at the following URL:

    https://<host>:<optionalPort>/design?tenant=system

  3. In the Engines panel, click Upload engine.

  4. In the file browser, select the downloaded engine ZIP file, and then click Open to upload the engine to the system tenant.

Naming engines

By default, when you upload an engine to your tenant, Exstream automatically assigns a name to the engine that matches its serialization. As an alternative, you can add user-friendly names—such as "Production" or "Development"—for each of the engines in your tenant. Then, when users need to select an engine for an engine run definition or a flow context, they can select from more intuitive options, instead of from a series of filenames.

Naming engines can also be useful if you need to replace an engine that is currently available in your tenant. For example, if you are in the process of phasing out an engine in your tenant, you can change the name of that engine to "Deprecated v20.2" so that users know not to select it in the future for their applications. After you have migrated all applications to other engines, you can delete that engine.

To name an engine:

  1. Sign in to the Exstream web client as an OTDS administrator user at the following URL:

    https://<host>:<optionalPort>/design?tenant=system

  2. In the Engines panel, select an engine from the list, and then click Open.

  3. Specify a name for the engine, as well as an optional description, and then click Save.

Setting a default engine

Setting the default engine for your tenant lets you specify the SBCS or DBCS engine version that applications will use if an engine is not otherwise specified. For example, you might configure the latest engine to be the default engine for your tenant so that new applications that use new functionality automatically use the latest engine version. Additionally, setting a default engine can save time when importing applications that already have a default engine specified. In that case, you don't need to configure the engine settings on those applications because they will automatically pick up the default engine for your tenant.

If you are working with a new deployment and have not yet uploaded any engines, the first SBCS and DBCS engines that you upload will automatically be marked as "Default" in System management view. So, in that scenario, the SBCS engine would be the default engine for SBCS applications, and the DBCS engine would be the default engine for DBCS applications.

Note: Whenever DocGen is installed or upgraded, the engine in the DocGen image is automatically set as the default. You can set another engine as the default after installation.

Remember that the default engine shows up as "Default" when users select between different engines in your tenant. So if you have an engine named "Development" and it is the default engine for your tenant, then that engine will show up as both "Default" and as "Development" in any lists where users select between engines.

Important: If you have applications that use a default engine, and you then select a different engine to be the default, behavior and output can change across those applications.

To set a default engine:

  1. Sign in to the Exstream web client as an OTDS administrator user at the following URL:

    https://<host>:<optionalPort>/design?tenant=system

  2. In the Engines panel, select an engine that is not currently marked as Default, and then click Open.

  3. Depending on the engine, select either the Default DBCS engine or Default SBCS engine check box, and then click Save.

Deleting engines

As you phase out specific engine versions, you can delete engines that are no longer needed in your tenant. Just remember that you cannot delete the default engine. If you want to delete the default engine, you must first set a different default engine for your tenant. Additionally, if you delete an engine that is being used by a run definition or flow context, then users will be alerted that they need to select a different engine when they open or try to use those objects later.

To delete an engine:

  1. Sign in to the Exstream web client as an OTDS administrator user at the following URL:

    https://<host>:<optionalPort>/design?tenant=system

  2. In the Engines panel, select an engine that you want to delete, and then click Open.

  3. Click Delete and then click Delete again to confirm deletion.

    The deleted engine no longer appears in the list of available engines.
