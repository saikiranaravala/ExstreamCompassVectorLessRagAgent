Creating and configuring domains

In an Exstream deployment, each tenant is a data-isolated environment. A domain in a tenant is simply a logical grouping of resources in the DAS asset library. By default, when you add or create a resource, it is associated with the domain in which it is created. However, all resources in a tenant—regardless of the domain with which they are associated—are stored in the same tenant schema in the database. A common scenario for using multiple domains is to separate resources for development, testing, and production. Domains can also be used to separate resources that are used for different business processes or geographical regions. The number of domains that you create in a tenant depends on your specific business needs.

When a user signs in to the Exstream web client, they must select a domain to work in. The Exstream role that is assigned to the user determines which domains they can see in Tenant View, and after they select a domain, they can see only those resources that are available to them within that domain. You use the Roles panel in Tenant management view to create roles and configure permissions for each role.

Creating a new domain

  1. In the Domains panel in Tenant management view, click Add.

  2. In the Properties panel, enter a name for the new domain in the Name box, and optionally, enter a description.

  3. If a tenant has both production and non-production keys associated with it, then choose one of the following options to specify which key the Exstream engine uses for this domain:

    • To use an available production key, select the Production check box.

    • To use an available non-production key, clear the Production check box.

    Important: Be sure to set this correctly when you create a new domain. You cannot modify this setting after you create the domain.

    Keep in mind that if the domain settings specify to use a key type that is not available in the tenant, the Exstream engine uses whichever key is available. The following table describes how the behavior of the engine changes depending on the key that is used.

    Key type Engine behavior

    Production key
    • Output (including preview and simulation) does not contain the demo watermark.
    • Transaction count is incremented for qualified batch engine runs.

    Non-production key
    • Output (including preview and simulation) contains the demo watermark.
    • Transaction count is not incremented for qualified batch engine runs.

    For more information about Exstream product keys, see Installing product keys for tenants.

  4. If you want Exstream to track the resources used for each job in the domain, select the Job tracing check box.

    When enabled, this option traces the flow instances that correspond to engine orchestration jobs. For each flow instance, it tracks which resources—and which version of those resources—were used, as well as who modified and approved them.

  5. Click Create.

Assigning roles to OTDS groups in a domain
  1. In the Domains panel in Tenant management view, select the domain for which you want to define user permissions.

  2. In the Group assignments tab in the Properties panel, click the Add group assignments button .

  3. In the Group assignments dialog box, use the following methods as needed to list available OTDS groups in the Groups area: 

    1. To list all available OTDS groups, click the button in the Search box.
    2. To list specific OTDS groups that are filtered by name, enter the name or part of the name in the Search box, and then click the  button.
    3. To list all OTDS groups in a specific partition, select the partition in the Partition list. Keep in mind that once you select a partition, you can switch between partitions but you can no longer see groups from all partitions at the same time.

  4. In the Groups area, complete the following steps to specify the group assignments for the current domain:

    1. In the Group ID column, select the OTDS groups for which you want to enable access to the current domain.
    2. In the Role column, for each selected OTDS group, click the Select roles list, and then select the roles that you want to assign to that group.
  5. Click Apply.
Modifying role assignments for an assigned OTDS group
  1. In the Domains panel in Tenant management view, select the domain for which you want to define user permissions.

  2. Go to the Group assignments tab in the Properties panel.

  3. In the Currently assigned groups area, click the Role list for the OTDS group for which you want to modify role assignments.

  4. In the Role list, select or clear role assignment check boxes as needed.
Removing an OTDS group assignment from a domain
  1. In the Domains panel in Tenant management view, select the domain for which you want to define user permissions.

  2. Go to the Group assignments tab in the Properties panel.

  3. In the Currently assigned groups area, click the Remove group assignment button for the OTDS group that you want to remove from the current domain.

Scheduling tasks for a domain

  1. In the Domains panel in Tenant management view, select the domain for which you want to add a scheduled task.

  2. In the Scheduled tasks tab in the Properties panel, click the Add scheduled task button .

  3. In the Scheduled task dialog box, specify a name for your task, then specify a cron expression and a type of action:

    Expire sorting assets

    This task expires the sort data and sort index files from a pre-sort run and those stored in the sorting repository. This is used to prevent outdated sorting assets from being processed. Once expired, the assets are not processed anymore.

    When you select this action, the following additional options become available:

    • Created—Specify the creation dates for the orchestration jobs that you want to process.

      • More than ... days before—Specify the number of days that an orchestration job was created before the scheduled task.

      • Fixed dates—Select a specific range of dates during which an orchestration job was created. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

    • Flow model—Specify the ID of the flow model in the pre-sort run that produced the sorting assets that should be expired.

    • Communication—Specify the ID of the communication in the pre-sort run that produced the sorting assets that should be expired.

    • Pre-sort queue name—Specify the name of the pre-sort queue in the pre-sort run that produced the sorting assets that should be expired.

    Mark jobs for deletion

    When you select Mark jobs for deletion, any qualifying jobs are tagged for removal but not removed from the database. When you select this action, the following additional options become available:

    • Start—The Start options let you specify the start dates for the orchestration jobs that you want to process.

      • More than ... days before—Specify the number of days that an orchestration job started before the scheduled task.

      • Fixed dates—Select a specific range of dates during which an orchestration job started. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

    • Finish—The Finish options let you specify the completion dates for the orchestration jobs that you want to process.

      • More than ... days before—Specify the number of days that an orchestration job completed before the scheduled task.

      • Fixed dates—Select a specific range of dates during which an orchestration job completed. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

    • Mode—The Mode options let you specify which types of orchestration jobs should be processed whenever the scheduled task runs.

      • Any—Select this option to process all jobs (batch and on-demand) that meet the criteria for the scheduled task.

      • Batch—Select this option to process only batch jobs that meet the criteria for the scheduled task.

      • On demand—Select this option to process only on-demand jobs that meet the criteria for the scheduled task.

    • Statuses—You can use the Statuses options to specify whether to process jobs that ran successfully or unsuccessfully as part of your scheduled task. Both options will be processed if you select neither check box or both check boxes.

      • Complete—Select this option to process only orchestration jobs that completed successfully.

      • Error—Select this option to process only orchestration jobs that encountered errors when they ran.

      • Canceled—Select this option to process only orchestration jobs that have been canceled.

    Service call

    Initiates an orchestration job run at a regularly scheduled interval. This is done by sending an HTTP request to a published HTTPS input channel. The HTTPS input channel must use the input selector processor as an after-event in order to process the body of the HTTP request properly. The body of the HTTP request contains the input channel used to retrieve the driver file for triggering the job and the relative path to the driver file. Optionally, the retrieved driver file can be cached for a specified number of days. The input selector processor uses this information to make the driver file available to the orchestration job.

    When you select this action, the following additional options become available:

    • Service name—Select the HTTP connector to associate with the task.

    • Input channel—Select the input connector used to retrieve the file in the input path.

    • Input path—The path to the file to retrieve; this is the relative path to what is defined in the input channel.

    • Caching—Specify the number of days for the caching timeout.

    Note: The Service name and Input channel lists are auto-populated with any published connectors in the flow context.

    To complete setting up a service call, follow these steps:

    1. Click Next.

    2. If you want to add one or more key-value pairs to the request header, click the Request headers button and fill out the following fields:

      • Key—Specify the key name for the request header element. This is the name of the orchestration system variable (created in Communications Orchestrator).

        Note: Keys in the header request are case-insensitive and must be unique. An error is issued if the user attempts to specify a duplicate key.

      • Value—Specify the value of the request header element. This is the value assigned to the orchestration system variable (created in Communications Orchestrator).

    3. Click Add.

    4. Click OK.

    Sorting & bundling

    This task is used to initiate the post-sort run for the sorting and bundling element. That is, the orchestration service picks sorting assets from the repository that are part of the pre-sort run, merges files, and sorts the assets according to the settings of the sort element. A post-sort run then produces the final documents.

    When you select this action, the following additional options become available:

    • Created—Specify the creation dates for the orchestration jobs that you want to process.

      • More than ... days before—Specify the number of days that an orchestration job was created before the scheduled task.

      • Fixed dates—Select a specific range of dates during which an orchestration job was created. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

    • Dry run—A dry run is typically used to test settings. If enabled, the processing state of sorting assets is not affected after the dry run, and thus the assets can still be picked up for additional post-sort runs.

    • Consolidate multiple applications—This setting configures if batch aggregation or application consolidation is used during the post-sort run. If enabled, multiple communications can be selected and are consolidated in the post-sort run.

    • Flow model—Specify the ID of the flow model that produced the sorting assets in the pre-sort run.

    • Communication—Specify the ID of the communication that produced the sorting assets in the pre-sort run. If Consolidate multiple applications is checked, you can select multiple communications.

    • Pre-sort queue name—Specify the name of the pre-sort queue that produced the sorting assets in the pre-sort run. This can be used to pick up pre-sort assets for a certain desired output format.

    • Post-sort queue name—Specify the post-sort queue name to use in the post-sort run. If you don't specify a post-sort queue name, then all post-sort queues will be included in the run.

    Update expiration time

    By default, all of the jobs, flow instances, and flow steps involved in batch (asynchronous) processing of orchestration flow are set to the earliest expiration time. The Update expiration time task is used to change that expiration. This option runs according to the cron expression that you specify and does not have additional settings associated with it.

    Wipe jobs

    This action removes from the database any qualifying jobs that have previously been marked for deletion. Additionally, wiping a job deletes any files and folders from NFS that are associated with the job. This option runs according to the cron expression that you specify and does not have additional settings associated with it.

    Wipe sorting assets

    This action removes any sorting assets from the database that have previously been marked for deletion.

    When you select this action, the following additional options become available:

    • Created—Specify the creation dates for the orchestration jobs that you want to process.

      • More than ... days before—Specify the number of days that an orchestration job was created before the scheduled task.

      • Fixed dates—Select a specific range of dates during which an orchestration job was created. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

    • Statuses—You can use the Statuses options to specify whether to process jobs that ran successfully or unsuccessfully as part of your scheduled task.

      • Processed—Select this option to wipe only the assets that have been processed.

      • Waiting—Select this option to wipe only the assets that are in a waiting state.

      • Expired—Select this option to wipe only the assets that are expired.

    • Flow model—Specifies the ID of the flow model in the pre-sort run that produced the sorting assets that should be wiped.

    • Communication—Specify the ID of the communication in the pre-sort run that produced the sorting assets that should be wiped.

    • Pre-sort queue name—Specify the name of the pre-sort queue in the pre-sort run that produced the sorting assets that should be wiped.

  4. Click OK.

  5. Click Save.

Supported cron expressions

Cron expressions for scheduled tasks must have five fields (minutes, hours, day of month, month, day of week). The task scheduler supports the following standard cron expressions:

Field Required Values Supported special characters
Minutes Yes 0-59 * , -
Hours Yes 0-23 * , -
Day of month Yes 1-31 * , -
Month Yes 1-12 or JAN-DEC * , -
Day of week Yes 0-6 or SUN-SAT * , -

Tip: You can combine slashes ( / ) with ranges to specify incremental values. For example, */5 in the minutes field indicates an event that happens every 5 minutes. Exstream does not support other non-standard characters for cron expressions.

Examples of cron expressions

The following table provides some examples of how cron expressions for scheduled tasks are executed based on the frequency of the task.

Scheduled task frequency Cron expression for scheduled task Expected result Actual behavior
Every 60 minutes (at minute 0) 0 * * * * Runs at minute 0 of each hour Runs at minute 0 of each hour
*/5 * * * * Runs every five minutes Runs once each hour, at minute 0, because the task runs only at minute 0
5 * * * * Runs every hour, at minute 5 Never runs, because the task runs at minute 0, not minute 5
Every 15 minutes (at minute 0, 15, 30, and 45) */15 * * * * Runs every 15 minutes Runs every 15 minutes
0 * * * * Runs at minute 0 of each hour Runs at minute 0 of each hour
*/20 * * * * Runs every 20 minutes, starting at minute 0 Runs once an hour, at minute 0, because the task does not run at minutes 20 and 40

Assuming that you use the default timing for your task (every 60 minutes, at minute 0), the following sample cron expressions might be useful when setting up scheduled tasks:

Cron expression Timing for scheduled task
0 2 * * 6 Runs at 2:00 a.m. every Saturday
0 2 * * 1-5 Runs at 2:00 a.m. every Monday-Friday
0 23 * * * Runs daily at 11:00 p.m.
0 2 1 * * Runs on first day of the month at 2:00 a.m.
0 1 */14 * * Runs twice a month, on day 14 and day 28, at 1:00 a.m.
0 1 1,15 * * Runs twice a month, on day 1 and day 15, at 1:00 a.m.
Configuring a domain to use the simple workflow

You can configure one or more domains in a tenant to use the simple workflow instead of the standard workflow. The simple workflow skips the Submitted for approval state. In a domain with simple workflow, if your user role can approve resources that are associated with the domain, you can approve those resources in one step. For more information about configuring user roles to approve resources, see Configuring roles and permissions.

You can change a the Workflow type property for a domain to Simple or Standard if the tenant-wide Use default workflow property for Content Author is enabled. When you add a domain, its Workflow type property is initially Standard.

Note: A domain's Workflow type property is only visible when the Use default workflow check box is selected.

When the Use default workflow check box is cleared, the default workflow in each domain, standard or simple, remains available in Communications Designer and in the asset library views.

If the Workflow type for a domain is Simple, the resources associated with that domain have the following workflow states and change options:

Current state Change options for simple workflow

Draft

Approve

Approved Make new draft

When you initially enable the simple workflow, some resources might be in the Submitted for approval or Rejected state. In these cases, your change options are as follows:

Initial state Change options for simple workflow

Submitted for approval

Approve or Return to draft

Note: In the simple workflow, Reject is not an option in the Submitted for approval state.
Rejected Return to draft
To configure a domain to use the simple workflow:

  1. Sign in to the Exstream web client as a tenant administrator user at the following URL:

    https://<host>:<optionalPort>/design?tenant=<tenantName>

  2. Select the Tenant management button.

  3. In the Tenant management view, enable the Use default workflow property for Content Author.

    Note: The tenant-wide setting Use default workflow affects Content Author only.

    The workflow in each domain, standard or simple, is always available in Communications Designer and in the asset library views.

  4. While in the Tenant management view, select the Domains panel and then select the domain in which to change the workflow type.

  5. In the Properties panel for the domain, select the Workflow tab.

    Tip: If you do not see the Workflow tab, repeat step 3 to enable the Use default workflow setting and then repeat step 4.

  6. In the Workflow type list, select Simple and click Save.

Deleting a domain

To delete a domain, select the domain in the Domains panel in Tenant management view, and then in the Properties panel, click Delete.

Note: To successfully delete a domain, make sure that the domain does not contain any resources and no roles are currently assigned to that domain.
