Understanding tenants and domains

When determining how you want to configure tenants and domains in your containerized Exstream deployment, consider the resource sharing needs in your organization, how transactions are counted, and the impact of each deployment approach on database management. These factors will determine how you create and manage tenants and domains.

A tenant is a data-isolated environment that users access to complete their daily work. Each tenant in a deployment has associated database schemas for each service in your deployment. The more tenants that you have, the more database schemas you will have to create and manage for the deployment, which requires more synchronization. This can introduce more manual steps into your process and be more difficult to maintain.

A domain is a logical grouping of the resources in the DAS asset library. You can have multiple domains in each tenant, all sharing the same database schema. Because all of the resources are contained within a single DAS database schema, resources can be associated with multiple domains in the tenant.

Choosing the right level of data isolation

You can choose to isolate resources in your deployment at the domain level or at the tenant level. If you isolate at the domain level, then all domains in the tenant can share resources and all use the same transaction counting. If you isolate at the tenant level, then resources are not shared between the domains in Tenant A and the domains in Tenant B. Instead, you must transport the resources between the DAS database in Tenant A and the DAS database in Tenant B. With tenant-level isolation, transactions are counted separately for each tenant.

At a high level, if your business requires strict data isolation between business groups—such as would be necessary in the healthcare or financial services industries—then you should use separate tenant environments for each business group to keep sensitive data completely separate. If, however, different groups in your Exstream environment need to collaborate closely as they work—such as with separate groups who are developing and testing the same applications—then you might use multiple domains within a single tenant to separate those groups. Whatever the scenario, Exstream provides the tools necessary to customize your deployment to meet your specific business needs.

Domain-level content isolation

Domain-level content isolation should be considered if all business units in your organization are able to access the same resources and use the same transaction counting. Because all domains have access to the same resources, it gives you the flexibility to easily make last-minute changes in production.

For domain-level content isolation, you can use a single DAS database to share resources across multiple business units in your organization. To isolate content at the domain level, create a tenant, add the required domains to that tenant, and then associate the resources in DAS with each of the domains.

For example, suppose that you want to isolate content between Dev, QA, and Prod for each logical business unit. Within a single DAS tenant, you can create three domains for each business unit, all tied to the same DAS database. Using the Exstream web client, you can associate the resources in DAS with each domain that uses the shared assets. Transaction counting is handled by the same OTDS tenant for all business units.

Tenant-level content isolation

Tenant-level content isolation should be considered when your deployment has any of the following requirements:

  • Access to DAS resources are restricted by business unit
  • Transactions are counted by business unit
  • Security policies differ by business unit

For tenant-level content isolation, you create a tenant for each business unit that requires isolation, which also requires you to create corresponding OTDS tenants, and DAS, Orchestration, and Empower (if needed) databases for each tenant in the deployment.

For example, suppose that you want to share resources for development and QA, but want each business unit to have security measures in place for all production content. In that scenario, you would create a development tenant that has separate development and QA domains for each business unit. You would then create a production tenant for each business unit, each with its own production domain.

One thing to note with this approach is that the more isolated your environments, the more manual steps you must complete to transport content from one tenant to another. OpenText does not recommend transporting content bi-directionally. That is, if you export content from a development tenant and import it into a production tenant, you should not import content from the production tenant back to the development tenant. Doing so can result in version conflicts for updated resources.
